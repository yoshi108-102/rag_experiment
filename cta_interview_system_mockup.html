CTAインタビューエージェント 要件定義書

参考文献: 21_433.pdf（認知タスク分析向けインタビュアエージェント研究）
本書は、研究内容を実装プロジェクト向けの一般的な要件定義書形式に再構成したドラフト。

1. 文書情報

文書名CTAインタビューエージェント 要件定義書
版数v0.1（初版）
作成日2026-03-06
作成目的開発着手前に要件の範囲・優先度・受入条件を明確化する
対象読者PO、開発、QA、運用、研究担当

2. 背景・目的

背景

CTAインタビューは有効だが、準備・実施・分析に工数がかかり、実施品質がインタビュア技能に依存する。

目的

半構造化インタビューを対話エージェント化し、自然さを保ちながら知識抽出の一貫性と再利用性を高める。

3. スコープ

対象範囲（In Scope）

  • テキストベースのCTAインタビュー実行
  • subject/topicに基づく文脈制御
  • 質問タイプ（STD/CDM/KA/RQ）の遷移制御
  • 相槌生成、回答解析、会話ログ保存
  • 知識抽出候補（構造化データ）の出力

対象外（Out of Scope）

  • 音声入出力・音声認識
  • 自動採点による人格/感情診断
  • 医療/法務等の専門助言の自動最終判断
  • 無制限な自由会話ボット機能

4. システム概要

構成要素

  • Interview Process Manager（状態管理）
  • Input Analyzer（キーワード・肯否疑問・分割節解析）
  • Question Selector（質問タイプ選択ルール）
  • Backchannel Selector（相槌タイプ遷移）
  • Response Composer（応答文生成）
  • Knowledge Extractor（知識候補抽出）

マスタ/辞書

  • 話題（subject）リスト
  • 質問対象（topic）リスト
  • 質問フレーズ辞書（22タイプ想定）
  • 相槌フレーズ辞書（9タイプ想定）
  • 生成方式設定（TEMPLATE_RANDOM / HYBRID_LLM）
  • CDMプローブ質問群
  • KA用過去回答・RQ用語句リスト

5. 業務フロー（論理）

① セッション開始

対象subjectを確定し、導入質問（STD1）を実行。

② 回答解析

キーワード、認知行動、肯定/否定/疑問、分割節数を更新。

③ 質問・相槌選択

状態変数と遷移ルールから次の質問タイプ/相槌タイプを決定。

④ topic/subject遷移

終了条件充足時に次topic、話題完了時に次subject、最終でSTD11。

注: 各topic内で質問を反復し、条件によりCDM/KA/RQ質問へ分岐する。

6. 機能要件

生成方式方針: Phase 1 は辞書ベースのランダム選択（品質安定・検証容易）を採用し、Phase 2 でLLM自然化を段階導入する。

ID 要件 優先度 受入条件（抜粋）
FR-01 セッション管理: 開始、進行、終了（STD11）を管理できること Must 開始時にsubjectを設定し、完了時に終了メッセージとログ確定が行われる
FR-02 文脈制御: subject/topic単位の進行順を管理できること Must topic完了条件を満たした場合に次topicへ遷移する
FR-03 質問タイプ遷移: 状態変数に基づき質問タイプを選択できること Must 肯否疑問/キーワード/分割節数を遷移判定に利用する
FR-04 相槌遷移: 質問回数・文脈に応じて相槌タイプを選択できること Should 新しい話段への遷移時は相槌なし応答を選択可能
FR-05 CDMプローブ: CDM質問をカテゴリ別に提示できること Must 情報獲得/状況認識/意思決定/行動カテゴリから質問を出せる
FR-06 Knowledge Awareness: 過去回答を参照した質問（KA1）を実行できること Should 類似topicの過去回答を引用し、比較質問を提示できる
FR-07 Ranking Question: 候補語句リスト提示と再深掘り（RQ1/RQ2）ができること Should 提示語句から利用者選択を受け、次質問対象に反映できる
FR-08 回答解析: 入力文の基本特徴量を抽出できること Must キーワード、肯否疑問、分割節数、認知行動分類を保持できる
FR-09 応答生成: 相槌 + 質問文の複文を生成できること（Phase 1: テンプレート、Phase 2: LLM自然化） Must Phase 1で辞書文面を返せること。Phase 2で文脈変数を保持した自然化文面を返せること
FR-10 ログ保存: ターン単位で状態変数と入出力を保存できること Must 後日分析用に会話履歴と遷移判断根拠を追跡できる
FR-11 設定変更: subject/topicや辞書を運用者が更新できること Could コード変更なしで対象領域を切替可能
FR-12 例外処理: 無回答・短回答・逸脱回答に対するフォールバックがあること Must 再質問、確認質問、終了確認のいずれかに遷移できる
FR-13 生成モード切替: テンプレート生成とLLM生成を機能フラグで切替できること Must セッション単位またはユーザー群単位で生成モードを切替できる
FR-14 LLMフォールバック: LLMエラー/タイムアウト時にテンプレート生成へ自動切替できること Must 失敗時も対話継続でき、fallback発生をログに記録する
FR-15 LLMガードレール: 選択済み質問タイプの意図を逸脱しないよう出力制約を適用できること Should 禁止表現・長文化・話題逸脱を検知した場合はテンプレート文面へ差し戻す

7. データ要件

エンティティ 主な項目 用途
in_memory_session_store session_id, user_id, subject, start_at, end_at, status インタビュー単位の進行・完了管理（プロセス内メモリ）
in_memory_turn_store turn_id, session_id, topic, question_type, backchannel_type, generation_mode, user_text, agent_text 対話履歴と遷移結果の記録（プロセス内メモリ）
in_memory_feature_store keywords, posnegq_flag, clause_count, cognitive_action_label 質問遷移ルール判定用の特徴量保持
in_memory_subject_topic_master subject_id, topic_id, display_order, active_flag 領域依存部分の管理（入替可能）
in_memory_phrase_dictionary phrase_id, type_group, type_code, text, locale 質問/相槌テンプレート管理
in_memory_knowledge_candidate_store cue, decision, action, difficulty, exception, confidence 抽出知識の構造化保存
in_memory_past_record_index topic_id, keyword, answer_excerpt, created_at KA/RQの候補提示材料
in_memory_generation_trace turn_id, model_name, prompt_version, latency_ms, fallback_used LLM導入時の品質・監査・コスト分析

MVPではDB接続を行わず、上記データは実行中メモリで保持する。必要時のみファイル出力で持ち出す。

9. 外部インタフェース要件

IF-ID 接続先 利用目的 備考

10. 非機能要件

ID カテゴリ 要件
NFR-01 性能 1ターン応答時間（質問選択+生成）を通常時5秒以内（目標P95）とする
NFR-02 可用性 セッション中断時に直前ターンから再開できること
NFR-03 保守性 subject/topic差し替えのみで新テーマへ展開できること
NFR-04 監査性 質問タイプ遷移の判定根拠（特徴量・ルール）を追跡可能にする
NFR-05 セキュリティ 個人情報を含む自由記述の保存時にマスキング/アクセス制御を適用する
NFR-06 ユーザビリティ 質問文は短文を基本とし、専門用語には補足を付ける
NFR-10 構成 MVPはLangChain + OpenAIを標準構成とし、生成処理の実装基盤を統一する
NFR-07 コスト Phase 2では1セッション当たりのLLM利用量上限を設定し、超過時はテンプレートモードへ切替える
NFR-08 再現性 Phase 1は同一条件で再現可能な選択ログ（乱数seed含む）を保持し、検証可能にする
NFR-09 品質管理 Phase 2はA/B比較で、自然さ指標がPhase 1を下回る場合に即時ロールバック可能とする

11. 制約・前提条件

  • 領域依存知識は subject/topic リストに閉じ込め、遷移ルール本体は共通化する。
  • MVPでは永続DBを導入せず、セッション/辞書/ログはインメモリ保持を前提とする。
  • 生成基盤は LangChain + OpenAI を標準とし、環境変数は `load_dotenv` で注入する。
  • 初期リリースは TEMPLATE_RANDOM を既定値とし、LLMは検証環境から段階導入する。
  • 外部APIの応答品質は一定でないため、固定テンプレート出力モードを恒久的に保持する。
  • 実運用では倫理審査・同意取得・ログ保管期間のルールを別紙で定義する。
  • 本書の数値目標は初期値であり、PoC結果で改定する。

12. 受入基準（初期案）

観点 基準
機能完備性 Must要件（FR-01,02,03,05,08,09,10,12,13,14）を全て満たす
遷移妥当性 任意の10セッションで、遷移不能/循環バグが発生しない
知識抽出性 回答分類で「情報が得られていない」比率がベースラインを上回らない
自然さ 主観評価アンケートで構造化固定質問方式に対して同等以上の評価を得る
運用性 管理者が画面操作のみでsubject/topicを追加し、反映できる
段階導入妥当性 Phase 1完了後にA/Bテスト条件を満たした場合のみPhase 2を本番有効化する

13. 未決事項・検討課題

  • 回答分類（タスク達成度）の自動判定基準をルールベースで行うか、モデル評価に寄せるか。
  • Phase 2移行判定に用いるKPI閾値（自然さ、抽出量、コスト）の確定。
  • KAで提示する過去回答の匿名化・要約粒度の運用方針。
  • RQ候補語句の生成頻度と品質管理手順。
  • 医療・防災等の高リスク領域でのガードレール設計。
